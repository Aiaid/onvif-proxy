# 07 — MCP Service Endpoint (Model Context Protocol)

Expose onvif-proxy's management capabilities (device CRUD, RTSP probing,
snapshots, ONVIF self-test, etc.) as MCP (Model Context Protocol) tools to AI
clients (Claude Code / Claude Desktop, etc.), so they can manage this proxy
directly — e.g. "add this RTSP stream as a virtual camera and run a
self-test on it."

- Implementation: the **official Go SDK** `github.com/modelcontextprotocol/go-sdk`
  (v1.6+, co-maintained with Google). This is the **sole exception** to the
  project's "third-party dependencies limited to yaml.v3" constraint, on the
  grounds that the MCP protocol surface (lifecycle/session/SSE/multi-version
  negotiation) is actively maintained long-term by the official library, so
  reimplementing it ourselves has low payoff; this exception has been
  synced into CLAUDE.md and docs/01/06.
- Transport: Streamable HTTP, a single `/mcp` endpoint, mounted on the
  existing Web UI mux (same port as the REST API), in **stateless mode**
  (`Stateless: true`, no `Mcp-Session-Id` issued). Protocol version
  negotiation and 405/session/SSE details are all handled by the SDK.
- Auth: reuses `withAuth` — when `web.username` is configured, site-wide
  Basic auth applies and `/mcp` is automatically covered as well.
- Go version: the SDK requires Go ≥ 1.25; `go.mod` and the Dockerfile build
  image were bumped in tandem (`golang:1.26-alpine`, see docs/05.en).

Client setup example:

```bash
claude mcp add --transport http onvif-proxy http://<host>:8080/mcp
# When web basic auth is enabled:
claude mcp add --transport http onvif-proxy http://<host>:8080/mcp \
  --header "Authorization: Basic $(printf '%s:%s' user pass | base64)"
```

## 1. Capability declaration

The `initialize` response is generated by the SDK; `serverInfo` is
`{name: "onvif-proxy", version: <Status().Version>}`; only the tools
capability is registered (no resources/prompts). `ServerOptions.Instructions`
is set to:

> onvif-proxy management server: wrap RTSP streams as virtual ONVIF cameras.
> Use list_devices/get_status to inspect, probe_rtsp/get_stream_info before
> adding a device, add_device/update_device/delete_device to manage,
> run_onvif_selftest to verify.

## 2. Tool set

12 tools, all registered on the same `mcp.Server`. JSON results are carried
in text blocks (as a serialized JSON string); snapshots are carried in image
blocks. Input structs are described with `jsonschema` tags, and the schema
is auto-generated by the SDK.

| Tool | Parameters | Result | Corresponding capability |
|---|---|---|---|
| `get_status` | none | Status JSON (version/advertise_ip/uptime_seconds/ffmpeg) | `Backend.Status` |
| `list_devices` | none | array of deviceView JSON (isomorphic to `GET /api/devices`) | `Backend.Devices` |
| `get_config` | none | raw config.yaml text | `Backend.ConfigYAML` |
| `apply_config` | `yaml` string required; `dry_run` bool default false | success: `"applied"` / `"valid"`; validation failure: isError + raw error text | `Backend.ApplyConfig` |
| `add_device` | `device` object required (same structure as the `POST /api/devices` body, see docs/04.en) | success: `"applied"`; validation failure: isError | reuses `buildDevice` + `ApplyConfig` |
| `update_device` | `uuid` string required; `device` object required | same as above; nonexistent uuid: isError | reuses edit logic (preserves uuid/mac/serial/info; empty auth password with unchanged username = password unchanged) |
| `delete_device` | `uuid` string required | same as above | reuses delete logic |
| `probe_rtsp` | `url` string required | rtsp.Result JSON | `rtsp.Probe` |
| `get_stream_info` | `url` string required | ffprobe stream-info JSON; missing ffmpeg: isError | same as `POST /api/test/streaminfo` |
| `take_snapshot` | `uuid` string required; `stream` string optional | image block (base64 JPEG); failure/no ffmpeg: isError | `Backend.Snapshot` |
| `run_onvif_selftest` | `uuid` string required | `[]onvifCheck` JSON (method/http_status/soap_fault/pass) | `runONVIFSelfTest` |
| `get_discovery_log` | none | WS-Discovery log JSON | `Backend.DiscoveryLog` |

Error semantics:

- **Invalid parameters / unknown tool** → SDK-level protocol error (the
  client receives a JSON-RPC error).
- **Business failure** (probe failure, config validation failure, missing
  ffmpeg, nonexistent uuid, etc.) → tool result `isError: true`, with the
  raw error text as content. In go-sdk, a handler returning a non-nil error
  is mapped directly to an isError result, and this mechanism is used as-is.

## 3. Implementation contract (internal/web)

New files `mcp.go` + `mcp_test.go`; two small refactors in `devices.go`;
REST behavior unchanged.

**`mcp.go`**:

- `func (s *Server) mcpHandler() http.Handler`: builds an `mcp.Server`
  (`mcp.NewServer(&mcp.Implementation{...}, &mcp.ServerOptions{Instructions: ...})`),
  registers all tools from §2 via `mcp.AddTool`, and returns
  `mcp.NewStreamableHTTPHandler(func(*http.Request) *mcp.Server { ... }, &mcp.StreamableHTTPOptions{Stateless: true})`.
- Uniform tool handler shape: input is a struct with `json`/`jsonschema`
  tags; text results are wrapped with the small helper
  `mcpJSON(v any) *mcp.CallToolResult` (strings passed through as-is,
  everything else via `json.Marshal`).
- Every per-device tool (take_snapshot / run_onvif_selftest / update /
  delete) first calls `s.findDevice(uuid)`, returning an error ("no
  configured device with that uuid") if it doesn't exist.
- **The API is authoritative as output by
  `go doc github.com/modelcontextprotocol/go-sdk/mcp <Symbol>`** (AddTool's
  generic signature, CallToolResult/Content fields, StreamableHTTPOptions)
  — do not write it from memory.
- If the SDK does not build in Origin validation (to prevent DNS rebinding,
  a MUST item in the MCP spec), wrap it at the mount point: when the
  `Origin` header is present and its hostname is neither loopback
  (localhost/127.0.0.1/::1) nor equal to the request `Host`'s hostname,
  respond 403; requests without an `Origin` header pass through.

**`web.go` registration** (in `New`): `mux.Handle("/mcp", s.mcpHandler())`
(no method restriction; GET/DELETE semantics are handled by the SDK).

**`devices.go` refactor**:

- Extract `func (s *Server) deviceViews() []deviceView`, shared by
  `handleDevices` and `list_devices`.
- Extract `func (s *Server) applyParsedConfig(cfg *config.Config) error`
  (marshal + `Backend.ApplyConfig`); the original `applyConfig(w, cfg)` is
  changed to call it; MCP's add/update/delete_device also call it.

Concurrency safety: all config changes go through `Backend.ApplyConfig`
(the manager inside main holds a lock); MCP introduces no new write path.

**`mcp_test.go`** coverage (via `httptest` + the SDK client
`mcp.NewClient` + `mcp.StreamableClientTransport`, over real HTTP):

- initialize succeeds, serverInfo.name = "onvif-proxy";
- tools/list returns exactly 12 tools, all names present;
- `list_devices` / `get_status` return results consistent with the fake
  Backend;
- `apply_config` dry_run validation failure → isError;
- `delete_device` with an unknown uuid → isError;
- calling an unknown tool → error;
- a bare `GET /mcp` (no session) does not panic, returns 4xx;
- a POST with a malicious `Origin` → 403 (if the wrapper layer is
  implemented).

## 4. Acceptance

1. `go build ./... && go vet ./... && go test ./...` all green; `go.mod`
   gains only the go-sdk and its transitive dependencies.
2. After starting the service locally, connect with a real client:
   `claude mcp add --transport http onvif-proxy http://127.0.0.1:8080/mcp`,
   and confirm initialize completes, the 12 tools are listed, and
   `get_status` and `list_devices` can be called successfully.
3. The Web UI's existing REST API and page behavior are unchanged
   (regression: built-in ONVIF self-test all green).
